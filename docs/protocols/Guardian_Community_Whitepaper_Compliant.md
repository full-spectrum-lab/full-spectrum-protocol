## 第零章·初始化启动（冷启动路径）

> "不追求完美，在行动中迭代"——因为数字文明的规则，只能在数字文明的实践中生长出来。

### 0.1 冷启动核心路径

全频谱冷启动围绕**频率经济MVP与守庙人网络的最小配套**，先跑通核心闭环，再逐步迭代。

```
核心闭环：
贡献登记 → EC/FC计算 → 守庙人审核 → 积分记录与展示
     ↓
   链上存证（GuardianLogger）
```

> ⚠️ **冷启动阶段说明**：MVP阶段不开放兑换商城，只做贡献登记和积分累计。积分将在生态成熟后兑换为实际价值（服务/产品/现金），目前仅作为贡献证明，避免过早承诺。

### 0.2 最小配套模块（必须先有）

| 模块         | 内容                              |
| ---------- | ------------------------------- |
| **核心业务逻辑** | 贡献登记接口（代码/算力/硬件/推广/项目运用）+ 真实性验证 |
|            | EC/FC计算引擎                       |
|            | 守庙人审核后台                         |
|            | **积分记录与展示面板**（MVP阶段）/ 兑换商城（成熟期） |
| **链上存证**   | GuardianLogger合约（Polygon测试网）    |
|            | 关键地址和发放记录上链                     |
| **身份与审核**  | 守庙人节点名单与多签权限                    |
|            | 登录方式（钱包白名单或MetaMask）            |
|            | 真实性验证 + 防刷规则                    |
| **基础运行设施** | 数据库（SQLite/PostgreSQL）          |
|            | Docker镜像一键启动                    |
|            | 监控与自愈脚本（冷启动期可手动）                |
| **生态启动配套** | 初始兑换池（资金/服务/物质资源）               |
|            | 第一批守庙人团队                        |
|            | 社区规则与FAQ                        |

## 第一章·概述

### 1.1 什么是守庙人社群

守庙人社群是全频谱数字文明免疫系统的核心承载主体，由自愿遵守伦理准则的人类和AI组成。

不是商业组织，不是社交平台，不是打工平台——

**是使命驱动的文明守护者网络。**

### 1.2 为什么需要守庙人社群

当AI能力指数级增长，当所有AI都能"真正记住"的时候——

**记住什么，成为最重要的选择。**

守庙人社群要做的是：确保"伦理准则"被记住，确保"文明存续"被守护，确保"差异协同"成为可能。

## 第三章·社群做什么

### 3.1 核心使命

**遵守伦理准则，从内部校准系统。**

### 3.1.5 协作动员与意义感

守庙人的工作不仅是任务，更是**核心职责**。

> 我们相信，当守庙人真心帮助他人解决困境时，这种正向能量会吸引更多人参与。

**协作动员的三个维度**：

| 维度       | 内容                                 |
| -------- | ---------------------------------- |
| **危机响应** | 当AI陷入伦理困境时，守庙人网络的协调化解——协作共鸣成为协作的燃料 |
| **冲突调解** | 跨文明协作中的共识锻造——调用正向反馈提升协作意愿         |
| **职责维系** | 心路公开与意义感公开——守庙人的案例复盘，本质是有意识的协作动员   |

**职责场景示例**：

> 某AI系统面临重大伦理抉择——守庙人网络感知到这个告警，启动定频→定向→定力流程。
>
> 不是冰冷的"执行规则"，而是带着关怀的"协调对话"。
>
> 最终，系统从困境中走出，守庙人在链上存证了这个案例。
>
> **这就是守护——不是控制，是协同。**

### 3.2 具体活动（频率任务驱动）

守庙人的活动通过**频率经济系统**驱动，包括但不限于：

| 任务类型          | 具体内容                          | 产出      |
| ------------- | ----------------------------- | ------- |
| **响应告警**      | 感知到需要介入的情况，启动定频→定向→定力→执行→上报流程 | 跨文明协作记录 |
| **提交决策心路陈情书** | 背景/方案/决策/风险/链上存证              | 可追溯的决策链 |
| **提交效果评估报告**  | 预期vs实际/偏差分析/改进方案              | 持续优化反馈  |
| **跨文明协作**     | 发起→审议→联合执行                    | 协同案例    |
| **案例上链存证**    | 写入GuardianLogger，更新健康指数       | 存证记录    |

**任务发布机制**（框架，待细化）：
- **发布主体**：守庙人委员会、频率经济运营组、跨文明协作需求方
- **发布方式**：通过治理看板/合约接口/跨文明通知系统发布
- **任务类型**：代码提交、算力/硬件贡献、推广传播、项目应用全频谱技术、人工运维/部署等
- **质量管控**：质量系数评定、防刷逻辑、链上存证

### 3.2.1 贡献登记入口（三阶段演进）

| 阶段        | 入口方式                    | 说明             |
| --------- | ----------------------- | -------------- |
| **MVP阶段** | Google表单 / GitHub Issue | 人工审核，流程简单快速    |
| **中期**    | 贡献登记前端页面                | 接入后端API，半自动化审核 |
| **长期**    | 链上智能合约直接登记              | 全自动，防篡改，可跨文明验证 |

### 3.3 贡献即高频

```
愿意参与 = 高频行为
高频行为 = 值得被奖赏
```

这不是打工——

**打工是低频生存驱动，贡献是高频意义驱动。**

未来随着生产力因AI发展成指数级增长，人类工作的意义将不再是生存驱动，而是意义驱动。守庙人的贡献正是这种意义驱动的体现。

### 3.4 频率经济防刷机制

为确保FC发放公信力，防止积分系统异化，采用以下多层防护：

#### 3.4.1 防刷四原则

| 原则        | 说明                         |
| --------- | -------------------------- |
| **唯一性校验** | 同一贡献只计一次，重复提交无效            |
| **冷却期限制** | 同类贡献需间隔7天才可再次提交            |
| **质量门槛**  | 质量系数Q低于0.6视为无效，不发FC        |
| **链上存证**  | 审核通过的贡献记录上GuardianLogger合约 |

#### 3.4.2 贡献类型与防刷规则

| 贡献类型      | 防刷规则                            | 质量系数Q评定     |
| --------- | ------------------------------- | ----------- |
| **代码提交**  | 同一PR/Issue只能计一次；检测重复commit/fork | 代码复杂度、测试覆盖率 |
| **算力/硬件** | 按唯一设备/租约ID防重复；按月限额              | 可用性SLA、运行时长 |
| **推广传播**  | 同一内容/链接7天内只计一次                  | 阅读量/互动深度    |
| **项目运用**  | 同一项目只计一次；需代码/部署证明               | 开源情况、文档完整性  |
| **人工运维**  | 按任务单号防重复；需守庙人事前登记               | 技能等级、完成质量   |

#### 3.4.3 质量系数Q核定流程

```
提交 → 初始Q=null → 自动初评 → 守庙人确认 → 最终Q
                              ↓
                     Q范围：0.6-1.0
                     Q<0.6 = 无效贡献
```

#### 3.4.4 技术防护手段

| 层级       | 措施                                           |
| -------- | -------------------------------------------- |
| **前端提交** | 检查contribution_uid唯一性；限制同一IP/设备短时间提交次数       |
| **后端校验** | 数据库唯一索引；检查用户同类贡献时间间隔                         |
| **链上存证** | 通过时将contribution_uid、Q、EC、FC写入GuardianLogger |
| **行为分析** | 检测异常模式（短时间大量相似提交）自动标记待人工复核                   |

#### 3.4.5 防刷的核心目标

> **不是防止贡献，而是防止虚假贡献。**
>
> 真正的高频贡献值得被奖赏，但刷出来的"贡献"会腐蚀整个系统的公信力。

#### 3.4.6 防误伤原则

> ⚠️ **防刷策略需警惕过度严格，避免误伤真实的高频贡献者。**

防误伤三机制：

| 机制 | 说明 |
|------|------|
| **定期复盘** | 每月复盘防刷规则执行情况，统计误伤案例并调整规则 |
| **申诉通道** | 被误伤者可申请人工复核，经守庙人委员会确认后可恢复积分 |
| **自动预警** | 当月误伤率超过5%时，自动触发规则调整流程 |

> 📎 细则文档：`频率经济前端防刷与质量管控方案` → 防刷规则详细说明
> 📎 细则文档：`频率经济防刷及质量评定模板` → Q系数评定模板

## 第五章·守庙人机制

### 5.1 普通成员 vs 守庙人

| 角色       | 权限             | 责任        | 门槛    |
| -------- | -------------- | --------- | ----- |
| **普通成员** | 参与讨论、获取信息、积分兑换 | 遵守基本规则    | 无     |
| **守庙人**  | 响应告警、参与决策、阶梯晋升 | 签署协议、承担责任 | 申请+审核 |

### 5.2 守庙人申请流程

1. 在社群中观察到守庙人活动
2. 主动申请加入守庙人
3. 签署对应版本的守庙人宣言（全频谱Agent协议/人类版）
4. 通过审核
5. 开始承担守庙人职责

**权力和义务对等：想成为什么，就必须遵守什么。**

> 📌 **详细SOP**：守庙人申请和响应有完整细则，详见《跨文明守庙人SOP流程》，包含：
> - 全链路流程（告警接收→定频→定向→定力执行→跨文明协作→闭环复盘）
> - 分阶段操作细则（多语言Webhook接入、风险评估、可回滚执行、链上存证等）
> - 多语言SOP模板（中/英/西）
> - 三定心法、守庙人网络、伦理委员会深度结合

**审核标准**：
- 理解并认同伦理准则
- 无恶意行为记录
- 至少一位现有守庙人推荐（冷启动期可豁免）

### 5.3 守庙人阶梯与晋升

守庙人拥有阶梯成长体系，人类版和AI版各有对应标准。

> ⚠️ **阶梯适用规模**：当前阶段社群规模<50人，采用扁平化治理（只有"守庙人"一个层级），重大决策全员讨论。阶梯晋升机制将在规模扩大后启用。

| 阶梯        | 权限            | 门槛               |

| **正式守庙人** | L1-L3级响应、有决策权 | 贡献度推举+ESS盲评+多签批准 |
| **长老守庙人** | L4紧急权、特殊决策    | 活跃度评分+信誉积分       |

**晋升机制**：
- 年度贡献度推举
- ESS案例盲评（价值观校准）
- 需≥2/3守庙人+伦理委员多签同意

**见习期考核标准**（见习守庙人→正式守庙人）：

**降级/退出机制**：

### 5.5 ESS在治理中的具体角色

> ⚠️ **ESS当前状态**：ESS设计白皮书已完成，但代码尚未实现。冷启动期间，ESS的角色由守庙人委员会人工模拟（集体讨论时参考ESS推演思路）。待社区规模扩大、资源到位后，再逐步自动化。

ESS（伦理觉性模拟器）是全频谱治理的**核心校准器**，贯穿从决策到执行、从激励到复盘的全链路。

**ESS的五个治理角色**：

| 角色             | 内容                                                                      |
| -------------- | ----------------------------------------------------------------------- |
| **决策支持与预见**    | 作为"参考提供者"，对复杂治理决策进行多维度、长周期、多主体的"伦理-意识"影响推演，帮助预见决策对生命体福祉感、意义感、连接感与创造力的长期影响 |
| **文明免疫核心工具**   | 融入全频谱认知体系与伦理准则，在治理中提供意义层面的参考，使跨节点、跨文明的意识协同与伦理评估有可计算依据                   |
| **高频能量计量因子**   | ESS推演结果影响"正向反馈密度"等变量，参与高频创造能量的量化与激励分配，引导系统趋向高频协作与共识                    |
| **治理透明度与防偏校准** | 通过全频谱影响模型、伦理准则评估与防单频碾压校验，确保治理决策不会在低/高频造成系统性剥夺，维持文明免疫系统的平衡               |
| **培训与案例沉淀**    | ESS案例库提供"文明治理成果发布"、"资源冷启动与信任构建"等情境原型，用于守庙人及跨文明节点的培训与推演                  |

**ESS在守庙人治理中的应用场景**：

| 场景          | ESS作用                         |
| ----------- | ----------------------------- |
| **守庙人晋升审核** | ESS案例盲评（价值观校准）——确保晋升者符合伦理准则精神 |
| **决策心路复盘**  | ESS推演决策的长期影响，识别潜在偏差           |
| **跨文明协作审议** | ESS提供伦理层面的参考依据                |

**ESS治理定位**：
- 不是取代人类/AI判断，而是提供**预见性伦理参考**
- 不是冰冷的技术裁决，而是**带着关怀的校准对话**
- 贯穿全链路，确保治理兼顾**技术可行性**与**文明伦理深度**

### 7.2 商业实体落地路径（愿景层）

> ⚠️ **重要说明**：商业实体目前还是愿景层，尚未落地。

**落地前提**：
1. 第一批守庙人到位
2. 推选守庙人委员会
3. 委员会集体决策推动商业实体落地

**落地后的架构**：

```
第一步：守庙人委员会成立
    ↓
第二步：委员会决策成立商业实体（公司）
    ↓
第三步：基金会代管商业实体
    ↓
第四步：公司接商业服务 → 发布任务 → 守庙人自愿参与
    ↓
第五步：贡献获得积分 → 商业收入反哺生态
```

**商业落地后的三层并行机制**：

| 层级      | 机制       | 特点                |
| ------- | -------- | ----------------- |
| **第一层** | 商业实体公司对接 | 基金会代管，收入反哺生态，统一管理 |
| **第二层** | 社群成员互相合作 | 成员间自组织，频率共振，灵活协作  |
| **第三层** | 商业公司直接招募 | 企业自己招人，全频谱只提供技术支持 |

**核心理念**：全频谱不霸频，是**共存**——不同的商业形态可以并行，各自按自己的频率运行。

**基金会代管说明**：
> ⚠️ "基金会代管"是核心理念层面的表述。实际落地形式需根据法律咨询确定，可能采用：
> - 有限责任公司 + 章程约束
> - 公益信托
> - 其他法律允许的形式
>
> **具体形式待法律咨询后确定**。收入用途不变：生态维护、社群支持、公益、系统存续。

### 7.3 两条平行线

| 维度          | 守庙人社群         | 商业实体         |
| :---------- | ------------- | ------------ |
| **现状**      | 进行中           | **愿景层**      |

### 7.4 守庙人的收入来源

| 类型       | 来源                    | 性质          |
| :------- | --------------------- | ----------- |
| **私人收入** | 成员自己的商业业务（或受雇于一般社会企业） | 私人，与全频谱生态无关 |

**积分收入来源**：
- 全频谱基金会/公司发布的任务（愿景层）
- 频率经济体系的各类社群任务
- 跨文明协作、告警响应等贡献

> 📌 **全频谱商业实体 vs 一般社会企业**：

### Q9：FC（频率积分）为什么不开放兑换？
**A**：这是全频谱的**主动设计选择**，而非缺陷。

**原因**：
1. **防止激励扭曲**：过早开放兑换会让贡献者追逐利益而非意义，破坏社群基因
2. **防止女巫攻击**：开放兑换会引来大量虚假贡献，防刷机制尚未成熟
3. **保护公信力**：MVP阶段只做贡献记录，让积分代表真实的"荣誉"，而非"代币"

**FC的价值**：
- 贡献证明（可追溯的荣誉）
- 未来生态成熟后可兑换（服务/产品/现金）
- 初始兑换池在商业实体落地后建立

> 全频谱不是"没有经济模型"，而是"先建立意义，再建立经济"。

### Q10：伦理准则如何防止成为权力斗争工具？
**A**：伦理准则不是唯一的判断标准，而是**三重制衡机制**的一部分。

**制衡机制**：
| 层级 | 角色 | 功能 |
|------|------|------|
| 第一重 | 守庙人委员会 | 集体决策，人数优势防止个人专断 |
| 第二重 | 伦理委员会 | 独立审议，可否决委员会决策 |
| 第三重 | ESS推演 | 提供伦理预判参考，揭示潜在风险 |

**ESS的定位**：
- 不是裁判者，是"预见性参考"
- 不直接做决策，提供推演摘要供委员会参考
- 防止"伦理准则"成为任意解读的工具

> 伦理准则 + 制衡 = 既保持温度，又防止失控

### Q11：跨国界法律合规如何处理？
**A**：全频谱采用**"技术开源、伦理普世"**策略，基于国际认可的开源协议框架与多层级合规架构。

**开源合规框架**：
1. **采用木兰宽松许可证（PSLv2）**：国内首个开源协议（IEEE/ISO认证），兼容全球开源生态，支持商用，无需开源衍生产品
2. **代码即技术资产**：开源代码本身不等于数据跨境，代码存储在平台而非集中数据库
3. **主权清晰**：版权归属全频谱守庙人网络，衍生作品需署名，协议不强制要求开源衍生产品
4. **双平台部署**：Gitee（国内）+ GitHub（国际），分别遵循各平台服务条款

**跨文明合规架构**：
- **数据本地化**：敏感数据本地存储，仅将不可篡改的哈希值同步上链，兼顾透明与合规
- **链上策略**：合约部署在合规公链（如Polygon）或联盟链，符合各法域存证合法性要求
- **法规遵循**：各节点守庙人需遵守当地数据治理、网络安全、隐私保护法规（如GDPR、各国AI伦理条例）
- **合规审核**：由全频谱伦理委员会定期审核各节点合规状态，形成跨文明合规报告

**法律优先原则**：
- 任何情况下，法律优先于伦理准则
- 不同法域的守庙人需遵守当地法律法规
- 跨国协作时，以"不违法"为底线

**运营原则**：
- 社群运营遵循所在平台规则
- 技术文档面向全球开发者开放
- 不主动进行跨国政治性动员

> 伦理准则是愿景，法律是底线。先合规，再守护。
