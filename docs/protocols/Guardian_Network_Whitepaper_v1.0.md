### 3.4 Anti-Gaming Mechanisms

To ensure FC issuance credibility and prevent points system alienation:

#### 3.4.1 Four Anti-Gaming Principles

| Principle | Description |
|-----------|-------------|
| **Uniqueness Verification** | Same contribution counted only once, duplicate submissions invalid |
| **Cooldown Period** | Same type of contribution requires 7-day interval before resubmission |
| **Quality Threshold** | Quality factor Q below 0.6 considered invalid, no FC issued |
| **On-Chain Storage** | Reviewed contributions recorded to GuardianLogger contract |

#### 3.4.2 Anti-Gaming Rules by Contribution Type

| Contribution Type | Anti-Gaming Rules | Quality Factor Q Assessment |
|-------------------|------------------|----------------------------|
| **Code Submission** | Same PR/Issue counted once; detect duplicate commits | Code complexity, test coverage |
| **Computing/Hardware** | Prevent duplicates by unique device/lease ID; monthly limits | Availability SLA, runtime |
| **Promotion** | Same content/link counted once within 7 days | Views/interaction depth |
| **Project Application** | Same project counted once; requires code/deployment proof | Open source status, documentation |
| **Manual Operations** | Prevent duplicates by task order; requires Guardian pre-registration | Skill level, completion quality |
