# Subject Levels and Node Types

This guide explains two different dimensions that are easy to confuse:

- **subject level**
- **node type**

They are related, but they are not the same thing.

## 1. Subject level

Subject level answers:

> what kind of subject is participating?

Examples:

- individual person
- single AI agent
- team
- enterprise
- organization cluster
- industry body
- city or region
- state-level institution

This is a scale question.

## 2. Node type

Node type answers:

> how is this subject entering the Full Spectrum ecosystem right now?

Examples:

- external tool node
- compatibility node
- certified node

This is an access and governance posture question.

## 3. Why the distinction matters

A large subject does not automatically become a certified node.

A small subject does not automatically become an external-only node.

For example:

- a single internal enterprise helper could eventually become a certified node if its action consequences are serious enough;
- a large organization could remain only a compatibility node if it wants limited integration without full certification;
- a purely external tool node may be very useful but still not carry full protocol responsibilities.

## 4. Suggested reading rule

Think of the two dimensions together:

```text
Subject level = scale and governance scope
Node type     = protocol entry mode and accountability posture
```

## 5. Practical interpretation

### A. Small subject + light node type

Example:

- a local internal assistant used only for drafting.

Likely path:

- external tool node or compatibility node;
- limited audit requirements;
- low-consequence use.

### B. Enterprise subject + stronger node type

Example:

- an enterprise AI customer-service or payment assistant that can influence refunds, commitments, or financial consequences.

Likely path:

- compatibility node first;
- certified node later if stronger assurance is needed;
- stronger audit, identity, and responsibility requirements.

### C. Cross-organization subject + network-sensitive node type

Example:

- multiple companies exchanging audit-compatible AI governance records.

Likely path:

- cross-enterprise profile;
- stronger conformance and trace requirements;
- protocol-network questions become more important.

## 6. Why local-first still matters

A subject does not need to join a global protocol network before trying Full Spectrum-compatible governance internally.

Recommended practical order:

1. internal engine use;
2. cell-level declaration and boundary clarification;
3. compatibility profile;
4. stronger certification only if necessary.

## 7. Design principle

The protocol should avoid forcing all subjects into one heavy entry mode.

The better pattern is:

- let subjects start where they are;
- make boundary and responsibility clearer as consequence grows;
- increase certification and interoperability only when justified.

This keeps the ecosystem usable without making every adopter carry the heaviest governance burden on day one.
